This file orients you (Claude Code) to the repo. Read it before making changes.

## What this is
PrivateDesk MemoryAgent is a **per-principal private memory layer for AI agents**. Each
principal is an isolated unit with its own persistent memory; one principal's memory is
never reachable by another's. The reference build is a **legal** scenario: a law firm with
several **matters**, where the memory wall between matters is the *ethical wall*.

The same code runs two ways, selected by one env var:
- `DASHSCOPE_API_KEY` set  → Qwen Cloud (DashScope, Singapore, OpenAI-compatible endpoint)
- `DASHSCOPE_API_KEY` empty → local Ollama / open-weight Qwen3

## Commands
- `make dev` — build + run locally, wait for `/health`, seed the legal demo
- `make health` / `make seed` / `make logs` / `make test` / `make down`
- `make deploy IP=<public-ip> KEY=<path.pem> [USER=root]` — deploy to an Ubuntu ECS box
- Tests: `docker compose exec api pytest -q tests/test_isolation.py`

## Layout
```

  config.py                 all tunables; DASHSCOPE_API_KEY presence picks cloud vs local
  db.py                     SQLAlchemy models; _namespace() mints the crypto isolation key
  llm/{client,prompts}.py   LiteLLM wrapper + prompt composition
  memory/
    embeddings.py           text-embedding-v4 (1024-d)
    qdrant_client.py        THE isolation chokepoint — the only vector reader
    engine.py               write / recall / forgetting
  personas/__init__.py      one assistant persona per matter (litigation/employment/corporate)
  routers/{chat,members,inspector,actions,demo}.py
  tests/test_isolation.py   guard test: zero cross-namespace leakage
web/                        Next.js 14 cockpit (chat + inspector + ethical-wall view)
docs/                       diagrams, architecture doc, deploy runbook, demo script
scripts/                    dev-up, bootstrap-server, deploy-remote
```

## Invariants — do NOT break these
1. **The chokepoint is the only vector reader.** `memory/qdrant_client.py` `search()` must
   always be scoped to a single namespace. Never add a code path that reads vectors without a
   namespace filter — that is the whole security property. There is a guard test; keep it green.
2. **Namespaces are opaque crypto tokens** (`ns_` + `secrets.token_hex(16)`), generated by the
   `default=_namespace` on `Member.memory_namespace`. Never construct human-readable namespaces
   and never derive them from names/PII. The readable label lives in `Member.name`.
3. **The namespace is resolved server-side** from the principal record — never trust a
   client-supplied namespace.
4. **`NEXT_PUBLIC_API_BASE`** is baked into the web build; in a remote deploy it must be the
   reachable API URL (`http://<public-ip>:8000`), not localhost. The deploy script handles this.
5. **Never expose** Postgres (5432), Qdrant (6333), or Redis (6379) publicly. Only 8000 and 3000.
6. **Human-in-the-loop:** assistants draft; an attorney approves. Actions surface as proposed
   actions, not auto-executed.

## Honest state (don't overclaim)
- Login is a **dummy login (no passwords)** — but the auth is real: a signed token binds a session
  to one principal server-side and the API returns **403** for any other principal
  (`api/app/auth.py`). Real passwordless/SSO is a drop-in on top.
- Roles: `principal` (own data only), `supervisor` (cross-principal **metadata only** — never
  content), `demo` (the god-view for the isolation demonstration; labelled as such in the UI).
- Lean topology: **one principal = one namespace.** Team access to a shared principal and a
  client-facing view are extensions on top of the same engine.
- Two scenarios ship: **legal** (matters / ethical wall) and **healthcare** (patients /
  confidentiality), chosen at the login screen. Same engine — swapping domains is data, not code.
