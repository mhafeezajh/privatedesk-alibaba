# PrivateDesk MemoryAgent — Technical Architecture

How the system is built and how each component works. Companion to
[`DEMO-WALKTHROUGH.md`](DEMO-WALKTHROUGH.md).

---

## 1. What it is, in one paragraph

PrivateDesk MemoryAgent is a **per-principal private memory layer for AI agents**. Each
*principal* is an isolated unit with its own persistent memory; one principal's memory is never
reachable by another's. The wall between principals is enforced in the **retrieval layer** — not
merely asked for in a prompt — is bound to a **logged-in identity** (§3.5), extends to the **LLM
prompt cache** (§3.6), and is covered by a guard test. The same code runs against **Qwen Cloud
(DashScope)** or a **local Ollama** open-weight model, selected by a single environment variable.

**Two domains ship on the identical engine**, chosen at the login screen — swapping is *data, not
code* (`_SCENARIOS` in [`routers/demo.py`](../api/app/routers/demo.py)):

| Domain | Principal | The wall is | Confidential "needle" |
|---|---|---|---|
| **Legal** | a *matter* (`role: matter`) | the profession's **ethical wall** | Acme's $4.2M settlement ceiling |
| **Healthcare** | a *patient* (`role: patient`) | **patient confidentiality** | a patient's HIV status + medication |

## 3. The isolation model (the core property)

This is the whole product. Three rules make it true; the code enforces all three.

### 3.1 Opaque namespaces

Every principal (`Member`) is minted with a **namespace**: an opaque, 128-bit token generated
server-side in [`db.py`](../api/app/db.py):

```python
def _namespace() -> str:
    return "ns_" + secrets.token_hex(16)   # carries NO name / PII
```

- Generated by the ORM `default=_namespace` — it is never constructed from a name and never
  derived from PII. The human-readable label lives in `Member.name`, never in the namespace.
- Resolved **server-side** from the principal record on every request. A client never supplies
  a namespace, so it can't be spoofed.

### 3.2 The chokepoint — the only vector reader

[`memory/qdrant_client.py`](../api/app/memory/qdrant_client.py) is the **sole module allowed to
query Qdrant**, and its `search()` is **always** scoped to one namespace:

```python
async def search(namespace, vector, limit, active_only=True):
    hits = await _client.search(
        collection_name=_collection_for(namespace),          # per-namespace collection
        query_filter=_namespace_filter(namespace, active_only),  # + payload filter
        ...
    )
    for h in hits:
        assert h.payload["namespace"] == namespace, "ISOLATION BREACH"  # defense in depth
```

Two modes (env `ISOLATION_MODE`):
- **`collection_per_member`** (default) — one physical Qdrant collection per namespace
  (`mem_ns_…`). The stores are *literally* separate, which the Ethical-Wall view visualizes.
- **`single_collection`** — one collection, mandatory `namespace` payload filter.

In **both** modes the namespace payload filter *and* the post-query assertion run — belt and
suspenders. If a foreign point ever came back, the request fails loudly rather than leaking.

### 3.3 Prompt-level reinforcement

[`prompts.compose_system_prompt`](../api/app/llm/prompts.py) injects **only the selected
principal's** memories and instructs the model to refuse cross-context questions. This is the
*last* line, not the wall itself — the retrieval layer already made foreign memory
unreachable.

### 3.4 The guard test

[`tests/test_isolation.py`](../api/tests/test_isolation.py) asserts zero cross-namespace
leakage. It is the codified invariant; keep it green. Run it live:
`docker compose exec api pytest -q tests/test_isolation.py`.

### 3.5 Authentication & access control (who may read a namespace)

[`auth.py`](../api/app/auth.py) — a *dummy* login (no passwords) that nevertheless establishes a
**real, server-verifiable identity**. Tokens are HMAC-signed with `SESSION_SECRET` (stateless — no
store, survive restarts). Three roles:

| Role | May do | May **not** |
|---|---|---|
| `principal` (bound to one member) | read/write **only that principal's** content | anything else → **403** |
| `supervisor` | cross-principal **metadata** (`/inspector/overview`, `/report`) | read content or chat → **403** |
| `demo` | read across principals (the labelled isolation demonstration) | — |

`POST /api/auth/login` binds the token to a member **server-side**; `authorize_content()` /
`authorize_metadata()` enforce it on chat, memories, audit, maintenance, and actions. A client can
never widen its own access by passing another `member_id` — the check is server-side.

### 3.6 Cache isolation (the second-order leak)

Providers cache prompt **prefixes**, and all principals share one API key — so a shared cache is a
cross-principal **timing side-channel** (B can detect that A recently submitted a prefix). A cache
hit can't inject A's content into B's prompt, but membership inference alone breaks an ethical
wall. [`llm/cache_isolation.py`](../api/app/llm/cache_isolation.py) **partitions** the cache instead
of disabling it: an unguessable per-principal token (`HMAC(SESSION_SECRET, namespace)`) is
prepended to the cached prefix — cache hits *within* a principal (perf kept), never *across*
(isolation kept). `PROMPT_CACHE_MODE` = `partitioned` (default) | `off` | `shared`. Full threat
model: [`CACHE-ISOLATION.md`](CACHE-ISOLATION.md).

## 4. The memory engine

[`memory/engine.py`](../api/app/memory/engine.py) — three paths.

### 4.1 Write path (`write_memories`)

Runs **after** the reply is streamed, so it never blocks the user.

```
turn text ──► LLM extraction (JSON) ──► for each candidate (max 4):
                embed(content)
                search top-12 neighbours in this namespace (score-sorted)
                  └─ for each with cosine ≥ SUPERSEDE_SCAN_FLOOR (0.60):
                        ask LLM to classify OLD vs NEW:
                          ├─ "supersedes" → mark OLD superseded, store NEW   (stop)
                          ├─ "duplicate"  → skip, nothing new to store       (stop)
                          └─ "distinct"   → keep scanning, then store NEW
                persist row (Postgres) + upsert vector (Qdrant, wait=True) + audit
```

- **Extraction** ([`extraction_messages`](../api/app/llm/prompts.py)) pulls durable, reusable
  memories (`fact / preference / event / task / relationship`) with a salience score; skips
  chit-chat *and the user's own questions*; max 4 per turn.
- **Supersession** is how "forgetting" happens at write time: an update flips the old memory to
  `superseded` (struck-through in the UI) and links `superseded_by`.
- **Why a 0.60 floor + a 3-way LLM verdict.** `text-embedding-v4` similarities run low — a real
  update ("$4.2M ceiling" → "raised to $5.0M") scores only **~0.72**, so the original 0.92 cutoff
  meant supersession *never fired*. Lowering the floor alone would wrongly merge merely-similar
  memories, so the LLM makes the call three ways and "distinct" memories simply coexist. The
  evals caught this.

### 4.2 Read path (`recall`) — bounded, ranked

```
query ──► embed ──► qdrant.search(namespace, limit=K_CANDIDATES=64, active_only)
       ──► rerank each candidate by a blended score:
             blended = 0.60·similarity + 0.25·salience + 0.15·recency
       ──► keep top K_CONTEXT=6  ──► inject into system prompt
       ──► bump last_used_at on the used memories  ──► emit a trace
```

- **Why two K's.** `K_CANDIDATES` is how many vectors the reranker *sees*; `K_CONTEXT` is how
  many actually enter the prompt. On a 115-memory matter, a small candidate pool buries
  high-salience "needles" below the raw-cosine cutoff, so they never reach the
  salience-weighted rerank. `K_CANDIDATES` is set to **64** for exactly this reason (raised
  from the app default of 20 during hardening — see §9).
- **Recency** decays with a ~few-week half-life: `exp(-age_days / 30)`; never-used memories get
  a neutral 0.3.
- **The trace** (`candidates_considered`, `selected_count`, `approx_tokens`, per-memory
  `sim/sal/blended`) is what the Recall-trace tab renders — bounded recall made visible.

### 4.3 Forgetting (`run_maintenance`)

Exposed as the **Run maintenance** button. For the selected principal:
- **Expiry** — any active memory past `expires_at` → `expired` (in PG and Qdrant payload).
- **Salience prune** — `salience < SALIENCE_PRUNE_FLOOR (0.2)` **and** unused > 30 days →
  `expired`.

Superseded/expired memories keep their Postgres row (audit history) but are filtered out of
recall by the `active_only` search filter.

## 5. Request lifecycle — a chat turn end to end

`POST /api/chat` ([`routers/chat.py`](../api/app/routers/chat.py)) returns a **Server-Sent
Events** stream:

```
1. Resolve session → member (principal) → persona.        (server-side; namespace fixed here)
2. Persist the user message (Postgres).
3. RECALL: engine.recall(member, message)  →  bounded, namespace-scoped memories + trace.
4. Compose system prompt: persona + identity + ONLY this principal's memories.
5. Stream tokens from the LLM (LiteLLM → DashScope/Ollama):
      ── event: trace     … the recall funnel (inspector shows bounded recall)
      ── data:  {token}   … each token as it arrives
6. After the stream, in a fresh DB session (doesn't block the user):
      • persist the assistant message
      • WRITE PATH: extract + store new memories
      • HITL: ask the LLM "did the reply propose an action?"
              if yes → create a ProposedAction (status=pending)
      ── event: proposed_action   … surfaces Approve/Reject in the UI
      ── event: done
```

The SSE contract (`event: trace` → `data: {token}`… → `event: proposed_action?` →
`event: done`) is parsed by [`web/lib/api.ts`](../web/lib/api.ts) `streamChat` (a POST +
`ReadableStream` reader, since `EventSource` can't POST).

## 6. Components in detail

### 6.1 web — Next.js 14 cockpit
- **[`page.tsx`](../web/app/page.tsx)** — orchestrates principal selection, per-matter chat
  threads and sessions, and the inspector. Selecting a pill calls `/api/session/start`.
- **[`Chat.tsx`](../web/components/Chat.tsx)** — message stream + pending-action Approve/Reject.
- **[`Inspector.tsx`](../web/components/Inspector.tsx)** — Memory store (with **Run
  maintenance**), Recall-trace funnel, Audit log.
- **[`IsolationView.tsx`](../web/components/IsolationView.tsx)** — two matters side by side, each
  showing only its own active memories.
- **`NEXT_PUBLIC_API_BASE`** is baked at **build time**; a remote deploy must bake the public
  API URL (`http://<ip>:8000`), which the IaC/deploy script does automatically.

### 6.2 api — FastAPI
- **[`main.py`](../api/app/main.py)** — app factory, permissive CORS (demo), routers, and a
  `/health` that does **one real completion + one real embedding** to prove the provider is
  live (`llm_ok`, `embedding_dim_live`).
- **Routers:** `auth` (dummy login → signed identity, §3.5), `chat` (session + streaming),
  `members` (principal list + `require_member`), `inspector` (memories / audit / maintenance /
  **`report`** / **`overview`**), `actions` (HITL pending/approve/reject), `demo` (scenario seed).

### 6.7 The governance surface
The memory layer *is* the compliance surface, exposed two ways:
- **`GET /api/inspector/report?member_id=`** — per-principal governance report: memory state by
  status/kind, the full audit breakdown, HITL decisions, and an **attestation** statement.
  Metadata only (no memory text), so a supervisor may read it. Rendered by the **Governance** tab,
  with a one-click **Download report** (JSON).
- **`GET /api/inspector/overview`** — *supervisor/demo only*: the cross-principal dashboard —
  every principal's counts, isolation-block totals, and attestation, **and no content**. This is
  how oversight confirms the walls hold without seeing through them.

### 6.3 llm — the provider seam
[`llm/client.py`](../api/app/llm/client.py) wraps **LiteLLM** so cloud⇄local is a *config*
change, not a code change:
- `complete` / `complete_json` / `stream` — the only places model calls happen.
- `_with_retry` — exponential backoff on `429/5xx` (4 attempts).
- `_clean_model` strips the `dashscope/` or `ollama/` prefix for the OpenAI-compatible route.

### 6.4 embeddings
[`memory/embeddings.py`](../api/app/memory/embeddings.py) — `text-embedding-v4` (1024-d) via the
same seam. Validates the returned vector's shape and treats a degenerate/short vector as
retryable, so bad embeddings are never persisted (see §9).

### 6.5 Data stores
| Store | Role | Notes |
|---|---|---|
| **PostgreSQL 16** | System of record: `households, members, sessions, messages, memories, audit_log, proposed_actions` | Namespace is denormalized onto each `memories` row; `qdrant_point_id` links to the vector. |
| **Qdrant 1.12** | Vector store, one collection per namespace | Cosine distance, 1024-d. Only ever read through the chokepoint. |
| **Redis 7** | Provisioned for session/ephemeral state (`REDIS_URL` wired) | The reference build currently persists sessions in Postgres; Redis is in the topology for scale-out. |

### 6.6 personas
[`personas/__init__.py`](../api/app/personas/__init__.py) — one assistant voice per matter type
(litigation / employment / corporate), each with a system prompt and an `allowed_actions` list.
All share a common clause: *assist on THIS matter only, never finalize/send, defer to a
supervising attorney.*

## 7. Data model (essentials)

```
Household 1───* Member ───┬───* Session ───* Message
                          ├───* Memory        (namespace, kind, content, salience,
                          │                     status[active|superseded|expired],
                          │                     superseded_by, last_used_at, expires_at,
                          │                     qdrant_point_id)
                          ├───* AuditLog       (event_type, detail JSONB)
                          └───* ProposedAction (action_type, payload, status[pending|approved|rejected])
```

- `Member.memory_namespace` — the isolation key (`unique`, opaque).
- `Memory.status` drives forgetting; `active_only` recall filters on it.
- `AuditLog` is the compliance/inspector surface: `memory_write`, `memory_recall`,
  `isolation_block`, `memory_maintenance`, `hitl_approved|rejected`.

## 8. The cloud ⇄ local seam

One rule, one env var ([`config.py`](../api/app/config.py)):

```
DASHSCOPE_API_KEY set    →  Qwen Cloud (DashScope): qwen-plus / qwen3-max / text-embedding-v4 (1024-d)
DASHSCOPE_API_KEY empty  →  local Ollama:           qwen3:8b / qwen3:32b / nomic-embed-text (768-d)
```

Nothing else in the codebase branches on cloud-vs-local. Privacy isn't about *where the bytes
live* — it's isolation + open weights + never training on your data; this seam makes "same
architecture, two deployments" literally true.

## 9. Reliability hardening & how it's verified

Three automated suites gate every change — see [`TEST-CASES.md`](TEST-CASES.md) (engineer) and
[`FUNCTIONAL-TEST-CASES.md`](FUNCTIONAL-TEST-CASES.md) (business/UAT):

| Suite | What it proves | Current |
|---|---|---|
| [`evals/run_evals.py`](../evals/run_evals.py) | scores all four behaviors + HITL end-to-end; **exits non-zero on any isolation leak** | **100/100** |
| [`scripts/smoke-test.sh`](../scripts/smoke-test.sh) | auth matrix (401/403), both domains' walls, health | **12/12** |
| [`tests/test_isolation.py`](../api/tests/test_isolation.py) | vector-layer chokepoint over 120 memories | **1 passed** |

The evals are what *found* the bugs below (and the missing `isolation_block` audit event, and a
supersession that never fired) — they're a regression guard, not decoration.

Three fixes make the seed and recall deterministic (added after a first deploy surfaced a
half-indexed matter):

1. **Durable vector writes** — `qdrant_client.upsert(..., wait=True)`. Without it, a burst of
   115 upserts during seeding (while Qdrant is still warming up) could be acknowledged but never
   applied, leaving memories that exist in Postgres but are invisible to recall. This was the
   original root cause of a matter "not knowing" its own facts.
2. **Embedding validation** — reject and retry degenerate/wrong-dimension embeddings instead of
   persisting garbage.
3. **Post-seed self-heal** — [`demo._heal_member`](../api/app/routers/demo.py) counts the
   vectors that actually landed and re-embeds any memory missing one, reporting `vectors_healed`
   in the seed response. A healthy seed now reports `vectors_healed: 0`.
4. **`K_CANDIDATES` 20 → 64** — so high-salience needles on 100+ memory matters reliably reach
   the reranker (§4.2).

## 10. Deployment & Infrastructure-as-Code

Two paths, same stack.

### 10.1 Local
```bash
cp .env.example .env   # set DASHSCOPE_API_KEY (or leave empty for Ollama)
make dev               # build, wait for health, seed
```

### 10.2 Full IaC on Alibaba Cloud — [`infra/terraform/`](../infra/terraform/)

One `terraform apply` goes from nothing to a live, seeded demo:

```
VPC → vSwitch → Security Group → ECS (Ubuntu 24.04) → EIP
                     │                    │
      only 22/8000/3000 open       null_resource.deploy:
      (5432/6333/6379 closed)      ship source → render .env →
                                   docker compose up --build →
                                   wait /health → seed demo
```

| File | Role |
|---|---|
| [`network.tf`](../infra/terraform/network.tf) | VPC, vSwitch, security group — **only** 22/3000/8000 inbound |
| [`compute.tf`](../infra/terraform/compute.tf) | Ubuntu 24.04 ECS (`ecs.u1-c1m2.large`, 2 vCPU/4 GB), EIP, generated SSH key, power state |
| [`deploy.tf`](../infra/terraform/deploy.tf) | Package + ship source, render `.env`, build with correct `NEXT_PUBLIC_API_BASE`, start, seed |
| [`templates/env.tftpl`](../infra/terraform/templates/env.tftpl) | Server `.env` (DashScope key, session secret, `K_CANDIDATES`) |
| [`variables.tf`](../infra/terraform/variables.tf) / [`outputs.tf`](../infra/terraform/outputs.tf) | Tunables (region, size, CIDRs, `k_candidates`) / URLs, SSH, isolation-test command |

Design points:
- **Credentials never committed** — `ALICLOUD_ACCESS_KEY/SECRET_KEY` come from the environment;
  the DashScope key via `TF_VAR_dashscope_api_key`; Terraform state and the generated SSH key
  are git-ignored.
- **`NEXT_PUBLIC_API_BASE`** is baked as `http://<eip>:8000` — the #1 remote-deploy gotcha,
  handled automatically.
- **`source_hash` trigger** — `null_resource.deploy` re-runs on any app code change (a content
  hash of the shipped source), so `terraform apply` redeploys code without touching the box.
- **Idempotent** — re-applying keeps existing resources and re-runs only the deploy step.
- Commands: `make infra-up` / `infra-plan` / `infra-output` / `infra-down`.

## 11. Security posture & invariants

Do **not** break these (also in [`CLAUDE.md`](../CLAUDE.md)):

1. **The chokepoint is the only vector reader**, always namespace-scoped. No other code path
   reads vectors. Guard test stays green.
2. **Namespaces are opaque crypto tokens**, never human-readable, never derived from PII.
3. **The namespace is resolved server-side** from the principal record — never client-supplied.
4. **`NEXT_PUBLIC_API_BASE`** must be the reachable API URL in a remote deploy.
5. **Never expose** Postgres (5432), Qdrant (6333), or Redis (6379) publicly — only 8000/3000.
   The Alibaba security group enforces this externally, independent of the host.
6. **Human-in-the-loop** — assistants draft; an attorney approves. Actions are *proposed*, not
   auto-executed.

## 12. Honest state (what's real vs. next)

- **Login is a *dummy* login (no passwords)** — but the auth is real: a signed token binds a
  session to one principal server-side and the API 403s any other principal (§3.5). Swapping in
  real passwordless/SSO is a drop-in.
- **Lean topology:** one principal = one namespace. Team access to a shared principal and a
  client-facing view are extensions on top of the same engine.
- **Redis** is provisioned but the reference build persists sessions in Postgres.
- **DB migrations** are `create_all` for hackathon speed; swap for Alembic in production.
- **CORS** is fully open for the demo; lock it down for production.
